# My-Farm-App Architecture Plan

## Project Overview
This is a **Next.js 16** web application for Nigerian farmers to manage poultry/goat/pig farms. It uses:
- **Next.js 16** (App Router)
- **Supabase** (Backend/Auth/Database)
- **Google Gemini** (AI Consultant)
- **Tailwind CSS v4** (Styling)

---


## Key Issues & Architecture Problems

### 1. Duplicate Supabase Clients
- `lib/supabase.ts` - no env validation
- `utils/supabase.ts` - has env validation
- **Inconsistent usage**: `login` uses `@/lib/supabase`, `register/dashboard` use `@/utils/supabase`

### 2. Authentication Flow Problems
- `register/page.tsx` stores credentials in localStorage (insecure)
- `login/page.tsx` uses proper Supabase auth
- No session management - should use Supabase sessions

### 3. Unused/Orphaned Pages
- `app/setup-profile/page.tsx` - defined but not linked in the flow
- `app/login/page.tsx` - login page exists but register doesn't redirect to it after success

### 4. Dashboard Code Duplication
- Lines 225-258 in dashboard/page.tsx contain **duplicate header code** that was already rendered (lines 130-141)
- This appears to be copy-paste error leaving dead code

### 5. Hardcoded Business Logic
- Animal type limited to Poultry in landing page
- No multi-farm support
- Hardcoded Nigerian market assumptions

### 6. Missing TypeScript Types
- No TypeScript interfaces for: Farmer, DailyRecord, HealthLog
- All database operations use `any` types

### 7. Missing Error Handling
- No loading states for some operations
- Poor error feedback to users
- No form validation

### Architecture Patterns

1. **Single Supabase Client**: One `lib/supabase.ts` with proper env validation
2. **API Layer**: Functions in `lib/api/*.ts` handle all database operations
3. **Custom Hooks**: Encapsulate stateful logic (auth, data fetching)
4. **TypeScript Interfaces**: All data models typed
5. **Auth Middleware**: Protect routes via Next.js middleware
