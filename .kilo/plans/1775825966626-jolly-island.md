## NEW BUG: "User Already Registered" - STILL FAILING

**Problem**: Even with the getUser check, registration still fails with "User already registered" at line 127

**Root Cause Analysis**:
- `getUser(email)` at line 97 might not be finding existing users reliably
- OR there's a race condition between check and signup
- The error happens inside signUp AFTER the check passes

---

## Solution - Try signUp with Error Handling

Instead of checking first, try signUp and catch the specific error:

```typescript
// Try to create new user
const { error: authError } = await supabase.auth.signUp({
  email,
  password
})

if (authError) {
  // Check if error is "user already registered"
  if (authError.message?.includes('User already registered') || 
      authError.code === 'user_already_exists') {
    // Try signing in instead
    const { error: signInError } = await supabase.auth.signInWithPassword({
      email, 
      password
    })
    if (signInError) throw new Error('Account exists. Please try logging in.')
    // Sign in successful - continue to get/create farmer
  } else {
    throw authError  // Re-throw other errors
  }
}
```

**Key Difference**: Wrap signUp in try/catch and handle the specific error gracefully, rather than checking first which might not work reliably.

## BUG 4: Session Not Persisting / Registered Users See Registration Page

**Problem**: 
1. Users who registered see registration page again on refresh
2. Users get logged out unexpectedly

**Root Cause**: 
1. **No localStorage fallback** in checkSession - if Supabase session check fails, no fallback
2. **Phone format mismatch** - session extracts `+234` prefix but farmer might be stored without it
3. **Middleware runs but auth cookies might not work** - server-side check might fail

**Expected Flow**:
- User registers → redirected to setup-profile → redirected to dashboard
- User refreshes page → session restored from Supabase → stays on dashboard
- User tries to access /register while logged in → redirected to dashboard

**Current checkSession Issue (line 19-38)**:
```typescript

if (session?.user?.email) {
  const phone = session.user.email.replace('@farmpulse.app', '')
  const farmer = await getFarmerByPhone(phone)  // FAILS if phone format doesn't match!
  if (farmer) {
    // success
  }
}
setState({ user: null, loading: false })  // Falls through to logged out state!
```

**Fixes Needed**:

### Fix 4a: Add localStorage fallback in checkSession
```typescript
const checkSession = useCallback(async () => {
  try {
    // First try Supabase session
    const { data: { session } } = await supabase.auth.getSession()
    
    if (session?.user?.email) {
      const phone = session.user.email.replace('@farmpulse.app', '')
      const farmer = await getFarmerByPhone(phone)
      
      if (farmer) {
        setState({ user: farmer, loading: false, error: null })
        localStorage.setItem('farmer_id', farmer.id)
        return
      }
    }
    
    // FALLBACK: Check localStorage if Supabase session fails or farmer not found
    const storedFarmerId = localStorage.getItem('farmer_id')
    if (storedFarmerId) {
      const farmer = await getFarmerById(storedFarmerId)
      if (farmer) {
        setState({ user: farmer, loading: false, error: null })
        return
      }
    }
    
    setState({ user: null, loading: false, error: null })
  } catch {
    // On error, still try localStorage
    const storedFarmerId = localStorage.getItem('farmer_id')
    if (storedFarmerId) {
      const farmer = await getFarmerById(storedFarmerId)
      if (farmer) {
        setState({ user: farmer, loading: false, error: null })
        return
      }
    }
    setState({ user: null, loading: false, error: null })
  }
}, [])
```

**Key Changes**:
1. Add fallback to localStorage farmer_id if Supabase session fails
2. Add getFarmerById import and function export
3. Wrap in try/catch for robustness

Check `lib/api/farmers.ts` - should have getFarmerById function
